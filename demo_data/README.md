# Demo data and engineering fixtures

`model_artifacts/` is an explicit, non-scientific registry used by
`scripts/mvp_fixture_smoke.py` and integration tests. It contains a tiny marker
file, a pinned YAML output description, a model card, and the deterministic
fixture adapter reference needed to exercise the same immutable bundle path as
a real model.

This registry is intentionally separate from `model_artifacts/registry.yaml`:

- production/development startup keeps missing real checkpoints unavailable;
- the fixture must be selected explicitly by the smoke script or a test;
- every fixture run records schema-v3 bundle and execution provenance;
- generated masks are simulated and cannot support scientific or competition
  performance claims.

Run from the repository root:

```bash
python scripts/mvp_fixture_smoke.py
python scripts/mvp_fixture_smoke.py --state-dir ./tmp/fixture-mvp
```

## Browser acceptance assets

`acceptance/` contains the deterministic 2048 × 1536 public engineering image and corrected mask
used by the end-user walkthrough:

- `acceptance/nanoloop_ui_acceptance_fixture.png`
- `acceptance/nanoloop_ui_acceptance_corrected_mask.png`

They are generated by `scripts/generate_acceptance_fixture.py`, contain no private SEM pixels, and
are suitable for public screenshots. They validate the UI, model-runtime, review, query, and export
paths only; they are not ground truth and do not establish scientific accuracy.

`rag/sources/project_sample_context.md` is the matching project-authored knowledge note used to
exercise document licensing, FTS5 ingestion, citations, refusal, and mixed evidence. It is not an
external literature corpus or a scientific source for performance claims.

Exact digests and the regeneration command are documented in
[`acceptance/README.md`](acceptance/README.md). The full click-by-click workflow is in the
[user acceptance guide](../docs/USER_ACCEPTANCE_GUIDE.md).
