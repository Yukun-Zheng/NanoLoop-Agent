# FunASR Nano 语音转文字可行性 POC 记录

## 结论

姚承志完成了一个可运行的语音转文字探索：在 Windows 环境中启动本地 Gradio 页面，加载
`FunAudioLLM/Fun-ASR-Nano-2512`，能够上传音频并获得中文转写，聊天记录还报告了方言样例可识别。
这足以证明“后续给对话框增加语音输入”有可行技术路线，但目前只按**拓展功能 POC**归档，
不接入主流程，也不宣称已经达到离线交付或生产可用。

记录日期：2026-07-20。

## 已提供的证据

- Gradio 页面运行在 `http://127.0.0.1:7860`，支持上传音频并显示识别结果。
- Windows 终端显示 `python webui_futureai.py` 能启动服务并加载模型代码。
- 终端提示机器没有安装 FFmpeg，因此当前使用 `torchaudio` 读取音频。
- 终端同时显示 `trust_remote_code: True` 和远程模型代码加载提示。
- 聊天截图中报告普通话/方言测试成功。
- 提供者展示了约 3 MB 的 `Fun-ASR.zip`，以及单独的 `ffmpeg-8.1.2-full_build.7z`。
- 提供的安装说明列出以下主要依赖：
  `torch`、`torchvision`、`torchaudio`、`transformers==4.57.3`、
  `funasr==1.2.9`、`gradio==6.2.0`、`modelscope==1.33.0`。
- 模型下载命令指向
  `FunAudioLLM/Fun-ASR-Nano-2512`，本地目录为
  `checkpoints/Fun-ASR-Nano-2512`。

## 本次没有验收的内容

截至本记录建立时，两个压缩包及模型目录没有提供到仓库工作区，因此项目负责人尚未检查源码、
依赖锁定、模型文件、许可证、文件哈希或恶意代码风险。本记录只根据截图和聊天证据归档，不能替代
对真实交付包的复现验收。

以下能力均尚未被证明：

- 断网安装和断网冷启动；当前流程仍需要从 ModelScope 下载模型。
- 固定模型 revision、权重 SHA-256 和可再现的依赖环境。
- 普通话、方言、噪声、多人、长音频的 CER/WER 指标。
- CPU/GPU 的冷启动时间、实时率、峰值内存和显存。
- 麦克风实时采集、停止/取消、超时、并发和错误恢复。
- 模型与运行时代码的许可证可否随项目再分发。
- `trust_remote_code=True` 所加载代码的版本固定与安全审计。

## 归档与安全规则

- 不把模型权重、ModelScope 缓存、FFmpeg 压缩包、音频样本或虚拟环境提交到 Git。
- 后续收到原始压缩包时，先在隔离目录解压，检查文件清单和源码，再决定是否迁入仓库。
- 外部模型资产使用清单记录来源 URL、精确 revision、许可证、SHA-256、大小和获取日期。
- 若必须启用远程代码，固定模型 revision，审阅实际加载文件，并记录其 SHA-256；不得在服务启动时
  隐式下载或执行未固定的远程代码。
- 测试音频必须确认来源和授权，不得把包含个人隐私的原始录音作为公开 fixture。
- FFmpeg 只记录官方来源、版本和许可证；不把第三方整包直接混入仓库。

## 未来接入形态

主项目需要的是稳定的“音频转文字”能力，而不是复用当前 Gradio 页面。真正接入时应把 ASR 做成
独立、可替换的后端组件：

```text
麦克风/音频文件
      |
      v
前端录音与上传 ----> 后端 ASR 接口 ----> 本地 ASR provider ----> 纯文本
                                                 |
                                                 v
                                    对话框现有文字输入流程
```

建议的最小接口语义：

- 输入：受大小、时长、格式限制的音频文件；不要直接接受任意本地路径。
- 输出：`text`、可选语言、时长、运行耗时、模型 ID/version、警告。
- 失败：不支持的格式、音频过长、无语音、模型不可用和资源不足应返回可区分错误。
- 前端只负责录音、上传和把成功文本填入输入框；识别模型不在浏览器或 Streamlit 页面内加载。

## 再启动此功能的验收门槛

只有满足下面条件，才把状态从“POC”改为“候选接入”：

1. 原始代码包可审阅，并有最小 `README`、锁定依赖和许可证清单。
2. 模型名称、revision、权重/配置 SHA-256、下载方式和离线资产目录固定。
3. 在一台干净机器上完成断网冷启动和至少 10 条授权样例的重复测试。
4. 记录 CER/WER 或人工逐条正确性、实时率、冷启动、内存/显存与失败样例。
5. 通过后端 provider/接口测试；不依赖 Gradio 才能工作。
6. 确认何时删除原始音频、日志是否脱敏，以及最大时长/大小/并发限制。

在 A～D 主链路完成前，本 POC 保持冻结，不占用当前前端和后端开发排期。
